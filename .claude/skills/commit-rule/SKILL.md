# Commit 규칙

Commit 메시지는 **제목**, **본문**, **꼬리말**로 구성한다.

---

## 1. 제목

```
{{유형}} : {{제목}}
```

### 1-1. 유형
| 유형 | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 포맷팅, 세미콜론 누락, 코드 변경이 없는 경우, .scss 관련 수정 |
| `refactor` | 코드 리펙토링 |
| `test` | 테스트코드, 리펙토링 테스트 코드 추가 |
| `chore` | 빌드 업무 수정, 패키지 매니저 수정 |

### 1-2. 제목 형식

```
( {{모듈}} ) {{주제}} 외 {{n}}건
```

- **모듈**: commit이 진행되는 모듈 경로 (예: `:api:watermark:tnp`)
- **주제**: 변경 내용 요약. 유형에 해당하는 단어는 포함시키지 않는다.
- commit 안에 변경된 사항이 1건이면 `외 n건` 생략, 2건 이상이면 `외 {n-1}건` 추가

#### 예시
```
feat : (:api:watermark:tnp) mediainfo 호출 프로세스 개선 외 1건
fix : [서비스세팅] 수정 중 취소 오류 외 2건
```

---

## 2. 본문

'기능추가', '버그수정', '개발환경변경' 3개 섹션으로 분리하여 작성하며, task 정보를 포함한다.
해당 섹션에 내용이 없을 경우 `없음`으로 기재한다.


```
1. 기능추가
- 없음

2. 버그수정
- [ ] T042 [P] [US3] `infra/paid-service-statistics.ts` getList function 에서 기간 설정 오동작 수정 

3. 개발환경변경
- 없음
```

---

## 3. 꼬리말

관련 있는 GitHub issue id를 작성할 때 사용한다. 개인 프로젝트라 외부 이슈 트래커 연동은 없으므로, **대개 꼬리말 섹션 자체를 생략한다.** 로컬에서 GitHub 이슈를 쓰는 경우에만 사용한다.

| 키워드 | 용도 |
|--------|------|
| `Resolves` | 해결된 GitHub issue id |
| `See also` | 참고하거나 관련있는 GitHub issue id |

#### 예시
```
Resolves : #1
See also : #2, #3
```

## 전체 커밋 메시지 예시

```
feat : (:api:watermark:tnp) mediainfo 호출 프로세스 개선 외 1건

1. 기능추가
- [ ] T011 [P] mediainfo 호출 시 비동기 처리로 변경.
- [ ] T012 [P] `MediaRepository` 결과 캐싱 로직 추가.

2. 버그수정
- 없음

3. 개발환경변경
- 없음

See also: #2, #3
```

## implementer 서브에이전트 커밋 규칙

- **모듈 단위 커밋**: 태스크 구현 완료 시 모듈(또는 논리적 단위)별로 커밋한다.
- **push 금지**: 커밋만 수행하고, 원격 저장소 push는 하지 않는다. (push는 사용자가 직접 수행)
- **커밋 메시지 작성 시 HEREDOC 사용**:
  ```bash
  git commit -m "$(cat <<'EOF'
  feat : (모듈) 주제

  1. 기능추가
  - 내용

  2. 버그수정
  - 없음

  3. 개발환경변경
  - 없음

  꼬리말
  EOF
  )"
  ```
- 변경된 파일만 `git add`로 명시적으로 스테이징한다 (`git add .` 사용 금지).
