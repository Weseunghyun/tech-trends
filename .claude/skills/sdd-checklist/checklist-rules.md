# 체크리스트 작성 규칙

## 핵심 원칙 - 구현이 아닌 요구사항을 테스트하라

모든 체크리스트 항목은 반드시 요구사항 자체를 다음 기준으로 평가해야 합니다:
- **완성도**: 필요한 모든 요구사항이 존재하는가?
- **명확성**: 요구사항이 모호하지 않고 구체적인가?
- **일관성**: 요구사항이 서로 정렬되는가?
- **측정 가능성**: 요구사항을 객관적으로 검증할 수 있는가?
- **커버리지**: 모든 시나리오/엣지 케이스가 다루어졌는가?

## 카테고리 구조 - 요구사항 품질 차원별 그룹화

- **요구사항 완성도** (필요한 모든 요구사항이 문서화되었는가?)
- **요구사항 명확성** (요구사항이 구체적이고 모호하지 않은가?)
- **요구사항 일관성** (요구사항이 충돌 없이 정렬되는가?)
- **수락 기준 품질** (성공 기준이 측정 가능한가?)
- **시나리오 커버리지** (모든 플로우/케이스가 다루어졌는가?)
- **엣지 케이스 커버리지** (경계 조건이 정의되었는가?)
- **비기능 요구사항** (성능, 보안, 접근성 등이 명시되었는가?)
- **종속성 및 가정** (문서화되고 검증되었는가?)
- **모호성 및 충돌** (명확화가 필요한 것은?)

## 항목 구조

각 항목은 다음 패턴을 따라야 합니다:
- 요구사항 품질에 대한 질문 형식
- spec/plan에 작성된 (또는 작성되지 않은) 것에 집중
- 품질 차원을 대괄호에 포함 [완성도/명확성/일관성 등]
- 기존 요구사항 확인 시 spec 섹션 참조 `[Spec §X.Y]`
- 누락된 요구사항 확인 시 `[Gap]` 마커 사용

## 절대 금지

- ❌ "확인", "테스트", "검증", "체크" + 구현 동작으로 시작하는 항목
- ❌ 코드 실행, 사용자 액션, 시스템 동작에 대한 참조
- ❌ "올바르게 표시됨", "정상 동작", "예상대로 기능함"
- ❌ "클릭", "내비게이트", "렌더링", "로드", "실행"
- ❌ 테스트 케이스, 테스트 계획, QA 절차
- ❌ 구현 세부사항 (프레임워크, API, 알고리즘)

## 필수 패턴

- ✅ "[시나리오]에 대해 [요구사항 유형]이 정의/명시/문서화되어 있는가?"
- ✅ "[모호한 용어]가 구체적인 기준으로 정량화/명확화되어 있는가?"
- ✅ "[섹션 A]와 [섹션 B] 간 요구사항이 일관되는가?"
- ✅ "[요구사항]을 객관적으로 측정/검증할 수 있는가?"
- ✅ "[엣지 케이스/시나리오]가 요구사항에 다루어져 있는가?"

## 시나리오 분류 및 커버리지

- 다음에 대한 요구사항 존재 여부 확인: 기본, 대안, 예외/오류, 복구, 비기능 시나리오
- 시나리오 클래스 누락 시: "[시나리오 유형] 요구사항이 의도적으로 제외되었는가 아니면 누락되었는가? [Gap]"

## 추적성 요구사항

- 최소: 항목의 ≥80%가 하나 이상의 추적성 참조를 포함
- 각 항목은 참조: spec 섹션 `[Spec §X.Y]` 또는 마커: `[Gap]`, `[모호성]`, `[충돌]`, `[가정]`

## 콘텐츠 통합

- 소프트 캡: 원시 후보 항목 > 40개이면 위험/영향 기준 우선순위 지정
- 동일한 요구사항 측면을 확인하는 유사 중복 병합
- 저영향 엣지 케이스 > 5개이면 하나의 항목으로 생성

## 체크리스트 유형 예시

| 도메인 | 파일명 | 초점 |
|--------|--------|------|
| UX | `ux.md` | 시각적 계층, 접근성, 인터랙션 상태 요구사항 품질 |
| API | `api.md` | 오류 응답, 속도 제한, 인증, 버전 관리 요구사항 품질 |
