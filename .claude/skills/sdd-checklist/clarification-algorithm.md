# 의도 명확화 알고리즘

## 동적 질문 생성

최대 3개의 초기 맥락 명확화 질문을 도출합니다 (사전 제작된 카탈로그 없음). 반드시:
- 사용자의 문구 + spec/plan/tasks에서 추출한 신호에서 생성
- 체크리스트 콘텐츠를 실질적으로 변경하는 정보에 대해서만 질문
- `$ARGUMENTS`에서 이미 명확한 경우 개별적으로 건너뛰기
- 폭보다 정밀도 우선

## 생성 알고리즘

1. **신호 추출**: 기능 도메인 키워드 (예: auth, 지연시간, UX, API), 위험 지표 ("critical", "must", "compliance"), 이해관계자 힌트 ("QA", "review", "보안 팀"), 명시적 산출물 ("a11y", "롤백", "계약").
2. **클러스터링**: 신호를 후보 집중 영역으로 클러스터링 (최대 4개), 관련성 기준 순위.
3. **대상 및 타이밍 식별** (작성자, 리뷰어, QA, 릴리스) - 명시적이지 않은 경우.
4. **누락된 차원 감지**: 범위 폭, 깊이/엄격도, 위험 강조, 제외 경계, 측정 가능한 수락 기준.
5. **질문 구성** (다음 원형에서 선택):
   - 범위 세분화 (예: "X와 Y의 통합 접점을 포함해야 하나요?")
   - 위험 우선순위 (예: "이 잠재적 위험 영역 중 필수 게이팅 검사를 받아야 하는 것은?")
   - 깊이 보정 (예: "경량 사전 커밋 검토 목록인가요, 공식 릴리스 게이트인가요?")
   - 대상 프레이밍 (예: "작성자만 사용하나요, 아니면 PR 리뷰 시 동료가 사용하나요?")
   - 경계 제외 (예: "이번에 성능 튜닝 항목을 명시적으로 제외해야 하나요?")
   - 시나리오 클래스 갭 (예: "복구 플로우가 감지되지 않았습니다—롤백이 범위에 포함되나요?")

## 질문 포맷팅 규칙

- 옵션을 제시하는 경우 컬럼이 있는 간결한 테이블 생성: 옵션 | 후보 | 중요한 이유
- 최대 A–E 옵션으로 제한; 자유 형식이 더 명확한 경우 테이블 생략
- 사용자가 이미 말한 것을 다시 말하게 하지 마세요
- 추측성 카테고리 피하기

## 상호작용이 불가능한 경우 기본값

- 깊이: 표준
- 대상: 코드 관련이면 리뷰어 (PR); 그 외에는 작성자
- 집중: 상위 2개 관련성 클러스터

## 후속 질문

Q1/Q2/Q3 답변 후: 2개 이상의 시나리오 클래스가 불명확하면 최대 2개의 추가 후속 질문 (Q4/Q5)을 할 수 있습니다. 총 5개 질문을 초과하지 마세요.
