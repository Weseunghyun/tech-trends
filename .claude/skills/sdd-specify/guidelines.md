# sdd-specify 상세 가이드라인

## 명세서 생성 실행 흐름

1. 입력에서 사용자 설명 및(선택) 작업 ID 파싱
   비어있는 경우: 오류 "기능 설명이 제공되지 않았습니다"
   작업 ID가 없으면 기능 설명에서 slug를 생성합니다(외부 트래커 연동 없음)
2. 설명에서 핵심 개념 추출
   식별 대상: 행위자, 동작, 데이터, 제약조건
3. 불명확한 측면의 경우:
    - 맥락과 업계 표준을 기반으로 합리적인 추측을 합니다
    - 다음 경우에만 [명확화 필요: 구체적 질문]으로 표시합니다:
        - 선택이 기능 범위나 사용자 경험에 크게 영향을 미치는 경우
        - 다른 의미를 가진 여러 합리적인 해석이 존재하는 경우
        - 합리적인 기본값이 없는 경우
    - **제한: 총 최대 3개의 [명확화 필요] 마커**
    - 영향도에 따른 명확화 우선순위: 범위 > 보안/개인정보 > 사용자 경험 > 기술적 세부사항
4. 사용자 시나리오 및 테스트 섹션 작성
   명확한 사용자 플로우가 없는 경우: 오류 "사용자 시나리오를 결정할 수 없습니다"
5. 기능 요구사항 생성
   각 요구사항은 테스트 가능해야 합니다
   명시되지 않은 세부사항에는 합리적인 기본값을 사용합니다 (가정을 가정 섹션에 문서화)
6. 성공 기준 정의
   측정 가능하고 기술에 독립적인 결과를 생성합니다
   정량적 지표 (시간, 성능, 볼륨)와 정성적 측정 (사용자 만족도, 작업 완료율) 모두 포함합니다
   각 기준은 구현 세부사항 없이 검증 가능해야 합니다
7. 핵심 엔티티 식별 (데이터 관련 시)
8. 반환: 성공 (명세서 계획 준비 완료)

## 일반 지침

### 빠른 가이드

- **사용자가 무엇을** 필요로 하고 **왜** 필요한지에 집중하세요.
- 구현 방법은 피하세요 (기술 스택, API, 코드 구조 없음).
- 개발자가 아닌 비즈니스 이해관계자를 위해 작성합니다.
- 명세서에 포함된 체크리스트를 생성하지 마세요. 별도의 명령어로 처리됩니다.

### 섹션 요구사항

- **필수 섹션**: 모든 기능에 대해 반드시 완료해야 합니다
- **선택 섹션**: 기능과 관련 있는 경우에만 포함합니다
- 섹션이 해당되지 않는 경우 완전히 제거합니다 ("해당 없음"으로 남기지 마세요)

### AI 생성 시

이 명세서를 사용자 프롬프트에서 생성할 때:

1. **합리적인 추측을 하세요**: 맥락, 업계 표준, 일반적인 패턴을 사용하여 빈 부분을 채웁니다
2. **가정을 문서화하세요**: 가정 섹션에 합리적인 기본값을 기록합니다
3. **명확화를 제한하세요**: 최대 3개의 [명확화 필요] 마커 - 다음과 같은 중요한 결정에만 사용합니다:
    - 기능 범위나 사용자 경험에 크게 영향을 미치는 경우
    - 다른 의미를 가진 여러 합리적인 해석이 존재하는 경우
    - 합리적인 기본값이 없는 경우
4. **명확화 우선순위**: 범위 > 보안/개인정보 > 사용자 경험 > 기술적 세부사항
5. **테스터처럼 생각하세요**: 모호한 모든 요구사항은 "테스트 가능하고 명확함" 체크리스트 항목에서 실패해야 합니다
6. **명확화가 자주 필요한 영역** (합리적인 기본값이 없는 경우에만):
    - 기능 범위와 경계 (특정 사용 사례 포함/제외)
    - 사용자 유형과 권한 (여러 상충하는 해석이 가능한 경우)
    - 보안/규정 준수 요구사항 (법적/재정적으로 중요한 경우)

**합리적인 기본값 예시** (이에 대해 묻지 마세요):

- 데이터 보존: 해당 도메인의 업계 표준 관행
- 성능 목표: 별도 명시가 없는 한 표준 웹/모바일 앱 기대치
- 오류 처리: 적절한 대체 수단을 갖춘 사용자 친화적 메시지
- 인증 방법: 웹 앱의 경우 표준 세션 기반 또는 OAuth2
- 통합 패턴: 프로젝트에 적합한 패턴 사용 (웹 서비스의 경우 REST/GraphQL, 라이브러리의 경우 함수 호출, 도구의 경우 CLI 인수 등)

### 성공 기준 가이드라인

성공 기준은 다음을 충족해야 합니다:

1. **측정 가능**: 구체적인 지표 포함 (시간, 백분율, 개수, 비율)
2. **기술에 독립적**: 프레임워크, 언어, 데이터베이스 또는 도구 언급 없음
3. **사용자 중심**: 시스템 내부가 아닌 사용자/비즈니스 관점에서 결과 설명
4. **검증 가능**: 구현 세부사항을 모르고도 테스트/검증 가능

**좋은 예시**:

- "사용자가 3분 이내에 결제를 완료할 수 있습니다"
- "시스템이 10,000명의 동시 사용자를 지원합니다"
- "검색의 95%가 1초 이내에 결과를 반환합니다"
- "작업 완료율이 40% 향상됩니다"

**나쁜 예시** (구현 중심):

- "API 응답 시간이 200ms 미만" (너무 기술적, "사용자가 즉시 결과를 봅니다" 사용)
- "데이터베이스가 1000 TPS를 처리할 수 있음" (구현 세부사항, 사용자 대면 지표 사용)
- "React 컴포넌트가 효율적으로 렌더링됨" (프레임워크 특정)
- "Redis 캐시 적중률이 80% 이상" (기술 특정)
