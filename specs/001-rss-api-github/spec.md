# Feature Specification: AI·개발/기술 트렌드 종합 대시보드

**Feature Branch**: `feature/daily-trends-dashboard`  
**Created**: 2026-06-13  
**Status**: Draft  
**Input**: User description: "AI·개발/기술 트렌드 종합 대시보드. 무료 공개 RSS·API(GitHub Trending·AI랩 블로그·OpenAI Codex·Hacker News)를 매일 수집·요약해 공개 GitHub Pages 정적 대시보드로 보여준다. X는 phase 2(유료/수동). 자세한 결정·소스·아키텍처는 PROJECT_BRIEF.md 참조."

## Clarifications

### Session 2026-06-13

- Q: 핫토픽/화두 섹션의 "트렌드 점수"를 유지할지? (주식 대시보드의 정량 분석이 아니라 "오늘 어디서 많이 회자되나"의 단순 집계임) → A: 점수 그대로 유지 — 교차 출현 수 + HN 포인트/댓글에 근거한 트렌드 점수를 계산·표시(US2/FR-008 유지).
- Q: data.json에 며칠치 데이터를 보관할지? → A: 최근 30일 롤링 보관(일자 간 중복 제거·"어제 대비" 비교 가능, 30일 초과분은 폐기).
- Q: 동일 항목(중복)을 무엇으로 식별할지? → A: 정규화된 원문 URL(쿼리·추적 파라미터 제거 후 동일 URL이면 동일 항목).
- Q: 카테고리당 하루 표시(요약) 항목 수 상한은? → A: 카테고리당 약 10개(상위 항목만 추려 요약 비용·로딩 절제).
- Q: 자동 수집 루틴 실행 시각/기준 시간대는? → A: 매일 아침 KST 08:00경 1회, "당일" 일자도 KST(Asia/Seoul) 기준.

## 용어 정의 (Terminology)

산출물 파일명·카테고리 표준 용어. 모든 산출물(plan/data-model/contracts/tasks)이 동일 용어를 사용한다.

- **산출 데이터 파일**: 대시보드가 로드 시 읽는 파일은 `latest.json`(당일 스냅샷). 일자별 아카이브는 `YYYY-MM-DD.json`(최근 30일). 매니페스트는 `index.json`. (이 문서에서 "data.json"이라는 일반 표현은 모두 `latest.json`을 가리킨다.)
- **카테고리(탭) — 5개 고정**: `ai_labs`(AI 랩 동향), `github_trending`(GitHub Trending), `codex`(OpenAI Codex), `eng_blogs`(엔지니어링/기술 블로그), `hot_topics`(핫토픽/화두). `eng_blogs`는 phase 1에서 비어 있거나 AI 랩 피드 일부를 공유할 수 있다.
- **핫토픽 표시**: `hot_topics` 탭은 개별 항목이 아니라 여러 소스를 묶은 **HotTopic 집계**(트렌드 점수순)를 표시한다. Hacker News 원천 항목은 이 집계의 근거로 사용된다.

## User Scenarios & Testing *(mandatory)*

이 대시보드는 **혼자 보는 사용자(소유자)** 한 명을 주 행위자로 한다. 매일 자동으로 갱신되는 정적 대시보드를 폰·아무 브라우저에서 열어 "오늘의 기술 화두"를 빠르게 파악하는 것이 핵심 가치다.

### User Story 1 - 매일 자동 수집·요약된 트렌드를 대시보드에서 본다 (Priority: P1)

소유자는 아침에 공개 URL(GitHub Pages)을 열어, 전날~당일 수집된 AI·개발/기술 트렌드를 카테고리별로 한눈에 본다. 각 항목은 한글 요약과 원문 출처 링크를 가진다. 별도 로그인·앱 설치 없이 폰·PC 어디서나 동일하게 본다.

**Why this priority**: 이것이 제품의 존재 이유다. 수집·요약·표시의 종단(end-to-end) 흐름이 한 번이라도 동작하면 그 자체로 매일 쓸 수 있는 MVP가 된다. 나머지 스토리는 이 흐름의 품질·편의를 높이는 부가 기능이다.

**Independent Test**: 수집 루틴을 1회 수동 실행해 산출 데이터 파일(대시보드가 읽는 `latest.json` 및 일자 아카이브 `YYYY-MM-DD.json` — 용어 정의 참조)을 생성하고, 대시보드 HTML을 공개 URL(또는 로컬 동일 출처)에서 열었을 때 카테고리별 항목·한글 요약·출처 링크가 표시되면 통과. 단일 스토리만으로도 "오늘의 트렌드를 본다"는 가치를 전달한다.

**Acceptance Scenarios**:

1. **Given** 수집 루틴이 당일 정상 실행되어 `latest.json`이 갱신된 상태, **When** 소유자가 공개 대시보드 URL을 연다, **Then** 5개 카테고리(AI 랩 동향, GitHub Trending, OpenAI Codex, 엔지니어링/기술 블로그, 핫토픽/화두) 항목이 최신순으로 표시되고 각 항목에 한글 요약과 클릭 가능한 원문 출처 링크가 보인다.
2. **Given** 어제와 오늘 모두 수집이 성공한 상태, **When** 대시보드를 연다, **Then** 마지막 갱신 시각(수집 기준 일자·시각)이 화면에 표시되어 데이터 신선도를 확인할 수 있다.
3. **Given** 소유자가 폰 브라우저로 접속, **When** 대시보드를 연다, **Then** 별도 가로 스크롤 없이 카드/탭이 모바일 화면에 맞게 표시된다.

---

### User Story 2 - "뜨거운 화두"를 트렌드 점수로 식별한다 (Priority: P2)

소유자는 여러 소스에 걸쳐 반복 등장하거나 Hacker News에서 높은 포인트·댓글을 받은 주제를 "핫토픽"으로 묶어, 무엇이 지금 화제인지 우선순위로 본다.

**Why this priority**: 단순 수집 나열을 넘어 "오늘 무엇이 화두인가"라는 핵심 질문에 답한다. P1(표시) 위에 얹는 부가 가치이며, P1 없이는 의미가 없으므로 P2다.

**Independent Test**: 동일/유사 주제가 2개 이상 소스에 등장하거나 HN engagement가 높은 주제가 "핫토픽" 섹션에 점수와 함께 상위로 노출되는지, 트렌드 점수 내림차순 상위 N(=10)이 표시되는지로 검증한다. (점수 산식·토픽 그룹핑 정본은 research.md R2; 하드 임계값이 아닌 그날 토픽 대비 상대 점수+상위 N 방식.)

**Acceptance Scenarios**:

1. **Given** 수집된 항목 중 동일/유사 주제가 복수 소스에 출현, **When** 트렌드 점수가 계산된다, **Then** 해당 주제가 핫토픽 섹션에 점수와 함께 상위로 표시된다.
2. **Given** Hacker News 항목의 포인트·댓글 수가 수집됨, **When** 점수를 산출한다, **Then** 점수 산식이 실제 수집된 수치에 근거하며 임의 보정 없이 계산된다.

---

### User Story 3 - 개별 소스 실패에도 대시보드가 부분 갱신된다 (Priority: P3)

특정 RSS/API가 일시적으로 실패해도, 나머지 소스 결과만으로 대시보드가 갱신되고 어떤 소스가 실패했는지 알 수 있다.

**Why this priority**: 무인 자동화의 견고함을 보장한다. 기능 자체보다 운영 안정성에 대한 요구로, P1/P2 동작을 보호하는 역할이라 P3다.

**Independent Test**: 한 소스 엔드포인트를 의도적으로 실패(타임아웃/404)시킨 상태로 루틴을 실행해, 나머지 소스 항목이 정상 적재되고 실패 소스가 빈 값/표시로 격리되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 여러 소스 중 1개가 응답 실패, **When** 수집 루틴이 실행된다, **Then** 나머지 소스 결과로 `latest.json`이 갱신되고 전체 빌드가 중단되지 않는다.
2. **Given** 한 소스가 실패함, **When** 대시보드를 연다, **Then** 실패 소스는 비어 있거나 "데이터 없음"으로 표시되고 추정값으로 채워지지 않는다.
3. **Given** 어떤 항목(예: 장기 인기 레포)이 어제 수집되어 이미 표시된 상태에서 오늘도 동일하게 수집됨, **When** 수집 루틴이 실행된다, **Then** 해당 항목은 중복으로 식별되어 한 번만 노출되고 매일 새 항목처럼 누적되지 않는다(FR-011).
4. **Given** 스케줄된 수집 시각, **When** 사용자 개입 없이 루틴이 실행된다, **Then** 1일 1회 무인으로 수집·요약·산출물 갱신이 완료된다(FR-001).

---

### Edge Cases

- 모든 소스가 동시에 실패하면? → 직전 성공분 `latest.json`을 유지(덮어쓰지 않음)하거나, 명시적으로 "오늘 수집 실패"를 표시하고 추정 데이터를 만들지 않는다.
- 영어 원문 글의 요약은? → 한글로 번역·요약한다. 원문 제목/링크는 보존한다.
- 동일 항목이 매일 재등장(예: 장기 인기 레포)하면? → 중복을 식별해 같은 항목이 매일 새 항목처럼 누적되지 않도록 한다.
- 수집 항목이 0건인 카테고리는? → 빈 섹션임을 명확히 표시(숨김 또는 "항목 없음")하고 오류로 처리하지 않는다.
- 외부 소스가 비정상 데이터(빈 제목, 비정상 URL, 과도하게 긴 본문)를 주면? → 형식·길이를 검증해 안전한 값만 적재한다.
- 요약 본문이 매우 길면? → 표시·저장 길이를 제한한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 매일 아침 KST(Asia/Seoul) 08:00경 무인 자동으로 수집·요약 루틴을 1회 실행해야 한다(MVP 기준 1일 1회). "당일" 일자 판정도 KST 기준으로 한다.
- **FR-002**: 시스템은 다음 무료·무키 공개 소스(닫힌 9개 식별자)에서 항목을 수집해야 한다 — GitHub Trending(`github_trending`, 비공식 RSS), AI 랩 블로그(`anthropic`·`openai`·`deepmind`·`meta_ai`·`mistral`·`xai`, 공개/공식 RSS), OpenAI Codex(`openai_codex`, 공식 RSS), Hacker News(`hackernews`, 공개 API: 포인트·댓글). 구체 엔드포인트는 research.md R1.
- **FR-003**: 수집된 각 항목은 **원문 출처 링크를 반드시 포함**해야 한다(출처 없는 항목은 표시하지 않는다).
- **FR-004**: 시스템은 영어 원문 항목을 한글로 요약·번역해야 하며, 요약은 실제 수집 데이터에만 근거하고 없는 사실을 만들지 않아야 한다.
- **FR-005**: 시스템은 수집 결과·요약을 정적 데이터 파일(공개 repo에 커밋되는 JSON: 당일 `latest.json` + 일자 아카이브 `YYYY-MM-DD.json` + 매니페스트 `index.json`)로 산출해야 한다.
- **FR-006**: 시스템은 정적 대시보드(공개 GitHub Pages)에서 같은 출처의 데이터 파일을 읽어 카테고리별로 표시해야 하며, 로그인·API 키·CORS 우회가 필요 없어야 한다.
- **FR-007**: 대시보드는 카테고리(탭/섹션)로 다음 5개를 구분 표시해야 한다 — AI 랩 동향(`ai_labs`), GitHub Trending(`github_trending`), OpenAI Codex(`codex`), 엔지니어링/기술 블로그(`eng_blogs`), 핫토픽/화두(`hot_topics`). `eng_blogs`는 phase 1에서 비거나 AI 랩 피드를 일부 공유할 수 있다(용어 정의 참조).
- **FR-008**: 시스템은 여러 소스 교차 출현 및 Hacker News 포인트·댓글에 근거해 트렌드(화제) 점수를 계산하고, 핫토픽을 점수 내림차순 상위 N(=10)으로 표시해야 한다. 점수 산식·가중치·토픽 그룹핑의 정본 정의는 research.md R2를 따른다(실측 수치만, 임의 보정 없음).
- **FR-009**: 개별 소스 호출 실패는 격리되어 전체 루틴을 중단시키지 않아야 하며, 부분 성공 결과로 대시보드를 갱신해야 한다.
- **FR-010**: 시스템은 외부에서 받은 값(URL·제목·본문)을 사용 전 형식·길이를 검증하고, 결측은 빈 값/"데이터 없음"으로 두어 추정값으로 채우지 않아야 한다.
- **FR-011**: 시스템은 중복 항목을 **정규화된 원문 URL**(쿼리·추적 파라미터 제거 후 비교)로 식별해, 일자에 걸쳐 동일 항목이 매일 새 항목처럼 누적 노출되지 않도록 해야 한다.
- **FR-012**: 대시보드는 마지막 수집 기준 일자·시각(데이터 신선도)을 표시해야 한다.
- **FR-017**: 시스템은 카테고리당 하루 표시 항목을 약 10개(상위 항목 우선)로 제한해야 한다.
- **FR-018**: 시스템은 일자별 수집 결과를 최근 30일 롤링으로 보관하고 30일을 초과한 데이터는 폐기해야 한다(일자 간 중복 제거 및 "어제 대비" 비교의 근거로 사용).
- **FR-013**: 대시보드는 모바일 브라우저에서 가로 스크롤 없이 읽을 수 있어야 한다(반응형).
- **FR-014**: 시스템은 API 키·토큰을 산출물·repo·로그·표준출력 어디에도 노출하지 않아야 하며, 비밀값은 환경변수/스케줄러 시크릿으로만 주입해야 한다.
- **FR-015**: 자동(무인) 경로는 클라우드/해외 IP에서도 접근 가능한 소스만 사용해야 한다(데이터센터 IP 차단·로그인 필요 소스 제외).
- **FR-016**: X(트위터) 소스는 phase 1 자동 수집 범위에서 **제외**한다(phase 2에서 유료 공식 API 또는 수동/대화형 보강으로 다룸).

### Key Entities *(include if feature involves data)*

- **TrendItem (트렌드 항목)**: 한 소스에서 수집된 단일 항목. 속성 — 제목, 원문 URL(필수), 출처/소스 이름, 카테고리, 한글 요약, 원문 언어, 수집/발행 일시, (선택) 지표값(예: HN 포인트·댓글, GitHub 스타 증가).
- **HotTopic (핫토픽)**: 여러 TrendItem을 묶거나 단일 항목에서 도출한 화제 주제. 속성 — 주제명, 트렌드 점수, 근거가 된 소스/항목 목록(출처 링크 포함), 출현 소스 수.
- **Category (카테고리)**: 대시보드 탭/섹션. 속성 — 이름(AI 랩 동향, GitHub Trending, 엔지니어링/기술 블로그, 핫토픽/화두), 소속 항목 목록.
- **DashboardSnapshot (대시보드 스냅샷)**: 하루치 산출물 단위. 속성 — 수집 기준 일자·시각(KST), 카테고리별 항목 모음(카테고리당 약 10개), 소스별 수집 성공/실패 상태. 최근 30일치 스냅샷이 롤링 보관된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 소유자는 대시보드 URL을 연 뒤 5초 이내에 당일 카테고리별 트렌드 항목과 한글 요약을 볼 수 있다.
- **SC-002**: 표시된 모든 항목은 클릭 가능한 원문 출처 링크를 100% 포함한다(출처 없는 항목 0건).
- **SC-003**: 자동 수집 루틴은 1일 1회 무인 실행되며, 소스 중 1개 이상이 실패해도 나머지 결과로 대시보드가 갱신되는 비율이 100%다(전체 중단 0건).
- **SC-004**: 핫토픽 섹션은 트렌드 점수 내림차순으로 정렬되며, 점수는 실제 수집 수치(소스 출현 수·HN 포인트/댓글)로만 산출되어 임의 보정 항목이 0건이다.
- **SC-005**: 대시보드는 폰(모바일)과 PC 브라우저 모두에서 가로 스크롤 없이 읽을 수 있다.
- **SC-006**: 산출물·repo·로그 어디에도 API 키/토큰 평문이 노출되지 않는다(시크릿 노출 0건).
- **SC-007**: 영어 원문 항목의 요약은 한글로 제공되며, 원문 제목·링크가 보존된다.
- **SC-008**: 연속 2일 이상 동일하게 수집되는 항목은 대시보드에 1회만 노출되어, 동일 항목의 일자별 중복 누적이 0건이다.
- **SC-009**: 수집 루틴은 사용자 개입 없이 1일 1회 스케줄로 실행되며, 실행 시 수집 기준 일자·시각이 산출물에 기록되어 무인 갱신 여부를 검증할 수 있다.

## Assumptions

- 주 사용자는 소유자 1인이며, 다중 사용자·인증·권한 관리는 범위 밖이다(공개 읽기 전용 대시보드).
- 갱신 주기는 MVP에서 1일 1회(매일 아침 KST 08:00경)로 가정한다(향후 빈도 조정 가능).
- 데이터 보관은 최근 30일 롤링이며, 그 이상의 장기 이력 보관·검색은 범위 밖이다.
- 요약은 수집 루틴을 실행하는 에이전트 LLM이 직접 수행하며 별도 LLM API 키에 의존하지 않는 것을 우선한다.
- "엔지니어링/기술 블로그" 카테고리는 공개/생성 RSS 목록(AI 랩 블로그 피드 및 대기업 eng blog RSS)으로 채우며, 소스 가용성에 따라 구성이 달라질 수 있다.
- 트렌드 점수 산식의 구체 가중치는 설계 단계에서 정하되, 실제 수집 수치에만 근거한다는 원칙은 고정이다.
- GitHub REST(스타 증가) 보강은 토큰이 있으면 사용하되 토큰 없이도 핵심 흐름이 동작하는 것을 기본으로 한다.

## Dependencies

- 무료 공개 외부 소스의 가용성: GitHub Trending RSS(비공식), AI 랩 블로그 RSS(공개/생성), OpenAI Codex 공식 RSS, Hacker News 공개 API. 소스 중단·형식 변경 시 해당 카테고리가 비거나 폴백이 필요할 수 있다.
- 공개 GitHub repo 및 GitHub Pages 서빙.
- 매일 루틴을 실행할 스케줄 자동화 환경.

## Out of Scope (Phase 1)

- X(트위터) 인플루언서 자동 수집(유료·로그인 제약 → phase 2).
- 다중 사용자, 로그인/인증, 개인화, 알림 발송.
- 서버·데이터베이스·백엔드 API(정적 사이트로 충분).
- 과거 이력 장기 보관·검색(필요 시 후속).
